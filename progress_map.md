## Current Boundaries

- Simulation code should stay in plain C# services/data; Unity objects remain presentation/tooling until a presentation step is confirmed.
- Use official Minecraft/NeoForge/Create source in this repo only for the smallest relevant reference set before implementing confirmed gameplay behavior.
- Do not implement from assumptions, previous prototypes, or copied Java source.
- Do not commit copied Create assets; private copied assets belong only under `Assets/PrivateTemp/Create`.
- Keep verification and checkpointing lightweight: one relevant Unity run after code changes, concise result recording, one implementation checkpoint when requested by the step workflow.

## Reference Notes To Preserve

- Clean local reference mirrors now exist under ignored `References/`: `Minecraft-1.21.1-sources` for decompiled mapped vanilla Java, `Minecraft-1.21.1-resources` for extracted vanilla assets/data, `NeoForge-21.1.219-sources` for official NeoForge source/resources, and `MDK-1.21.1-NeoGradle` for the backing NeoGradle workspace and NeoForm artifacts.
- Source-backed Create spec notes now live under `Docs/Specs/Create`, with one short file per covered first-slice block or item. Future agents should read the relevant spec first, use its Ponder notes for the player-facing mental model when available, and then follow its source anchors when deeper behavior detail is needed.
- Scheduled ticks: Create often guards `scheduleTick` with `hasScheduledTick(pos, block)` and may pass priorities; the Unity queue already models duplicate checks and stable priority ordering.
- Block entities: Create `SmartBlockEntity` initializes on first tick, runs an immediate lazy tick, and lets attached behaviors participate in tick/lazy-tick/read/write/lifecycle callbacks; the Unity foundation mirrors that shape.
- Serialization: behavior data currently shares the parent block entity payload as deterministic string key/value data; no JSON/NBT/file save format is chosen yet.
- Items: Create registers items centrally in `AllItems.java`; most use default stack size, while selected items use `stacksTo(1)` or `stacksTo(16)`. Unity item stacks are intentionally immutable until inventories/transport need mutation boundaries.
- Item Vault: Create's per-block vault capacity defaults to 20 stacks and full vaults form controller-based multiblocks. Unity currently implements only one block of Item Vault storage; multiblock connectivity, manual restrictions, capabilities, and rendering are deferred.
- First vertical slice ids now match Create registrations: `create:creative_motor`, `create:shaft`, `create:belt`, `create:creative_crate`, `create:brass_funnel`, `create:item_vault`, and item `create:andesite_alloy`.
- Private Minecraft grass textures are optional and uncommitted. The presenter now uses `grass_block_top.png`, `grass_block_side.png`, `grass_block_side_overlay.png`, and `dirt.png` for the flat world surface when those files are present under `Assets/PrivateTemp/Minecraft`, and otherwise falls back to the local `References/Minecraft-1.21.1-resources` mirror before using generated placeholder textures. `SampleScene` applies Minecraft's default grass tint (`GrassColor.getDefaultColor()`) to the top and side overlay instead of showing the raw grayscale top texture.
- Step 2 visual assets must read both Create resource roots: textures and some models from `src/main/resources/assets/create`, with many blockstates and generated item/block model JSONs from `src/generated/resources/assets/create`.
- Confirmed first-slice asset references exist for creative motor, belt textures, brass funnel texture, `andesite_alloy` item texture, and generated blockstates/models for shaft, belt, creative motor, brass funnel, Item Vault, shaft item, and belt connector item.
- The new Step 2.1 manifest preserves repo-relative source structure under the future private copy root, so `Assets/PrivateTemp/Create` can mirror both Create resource trees without embedding copied asset contents in committed code.
- First-slice item models split into two groups: `minecraft:item/generated` flat layers for `andesite_alloy` and `belt_connector`, and block-model parents for `shaft`, `creative_motor`, `creative_crate`, `brass_funnel`, and `item_vault`.
- Parented item models rely on inherited texture variables instead of standalone item geometry; `creative_crate` and `brass_funnel` specifically override parent texture keys rather than redefining the parent elements.
- `brass_funnel` uses `texture_size` `32x32`, while the other current first-slice item models stay on the default `16x16` texture grid.
- Some Create item elements use reversed `from`/`to` axes, so the Unity model reader must normalize element bounds before building face geometry.
- `creative_motor`, `brass_funnel`, and `item_vault` provide explicit `display.gui` transforms in the current first-slice references, while `shaft` and `creative_crate` fall back to default preview transforms.
- The local JSON readers are self-contained runtime code. They do not depend on `Newtonsoft.Json` or any newly-added Unity package, because the editor assembly surface in this project did not expose that dependency cleanly.
- Runtime-loaded Create textures used on 3D model faces must be point-filtered at load time. The flat item-card path already set point filtering on use, but the model-face path needs the shared loader to configure `FilterMode.Point`, clamp wrapping, and no anisotropy so the visuals stay pixel-sharp in `SampleScene`.
- The current block visual catalog intentionally excludes `create:belt`. Belt blockstates and model fan-out are larger enough to stay as their own follow-up slice.
- `create:item_vault` blockstate variants key on `axis`, while the current runtime block definition still uses `horizontal_axis`; Step 2.4 keeps the catalog on exact reference property strings and defers runtime aliasing/state-driven mapping.
- `create:brass_funnel` blockstates depend on `extracting`, `powered`, and `waterlogged` in addition to `facing`; the current catalog supplies those exact visual properties directly and defers authoritative gameplay-state wiring.
- Step 2.5 adds the first runtime-state bridge for world visuals: shaft, creative motor, and creative crate serialize directly from authoritative runtime properties; brass funnel currently fills the missing visual-only flags with Create's default `false` values; Item Vault maps runtime `horizontal_axis` to the blockstate's `axis` key while preserving `large`.
- Create's creative motor visual is not only the blockstate model body. The official block entity registration adds `AllPartialModels.SHAFT_HALF` as a separate oriented rotating visual, so a faithful Unity presentation needs both the motor body model and a separate rotating half-shaft on the facing side.
- The current focused kinetic slice uses the same narrow Create rules as the real motor/shaft connection: the Creative Motor outputs only on its `facing` side, its rotation axis is the facing axis, and straight shafts relay that same signed speed only when their `axis` matches the connection axis.
- The current shaft and motor animation speed uses Create's renderer convention: angular motion is derived from kinetic speed as `speed * 3/10` degrees per tick, which is `speed * 6` degrees per second at `20 tps`. A default Creative Motor speed of `16` therefore renders at `96 degrees/second`.
- The current shaft phase offset also follows Create's parity rule for plain rotating shafts: positions whose non-rotation axes sum to an even number start with a `22.5 degree` offset; otherwise the offset is `0`.
